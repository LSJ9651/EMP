\# 能耗分析工作流设计报告



> \*\*文档版本\*\*：v1.0

> \*\*编制日期\*\*：2026-06-17

> \*\*关联项目\*\*：能耗智能管理优化平台

> \*\*设计目标\*\*：构建基于扣子平台的能耗分析智能体工作流，实现设备能耗数据的自动异常检测与节能建议生成





\## 一、 业务背景与需求



\### 1.1 业务场景



在工厂能耗管理场景中，能源管理员需要\*\*定期分析设备运行数据\*\*，识别能耗异常、发现节能机会。传统方式依赖人工查看报表，效率低、难以发现隐蔽问题。



\*\*能耗分析智能体\*\*解决的核心问题：



| 业务痛点 | 智能体解决方案 |

|---------|-------------|

| 能耗数据量大，人工难以全面覆盖 | 自动遍历分析周期内所有数据点 |

| 异常模式隐蔽（如渐进式劣化、间歇性空转） | 通过统计学算法（3σ准则）自动检出 |

| 分析结果依赖个人经验，标准不统一 | 基于数据驱动的标准化输出，结果可复现 |

| 发现异常后缺少行动指引 | 大模型结合上下文生成具体可执行的节能建议 |



\### 1.2 功能需求



| 需求编号 | 需求描述 | 优先级 |

|---------|---------|--------|

| FR-01 | 支持单设备能耗数据智能分析 | P0 |

| FR-02 | 自动检测功率异常波动（跳变、骤降） | P0 |

| FR-03 | 自动识别持续低负载/空转状态 | P0 |

| FR-04 | 生成结构化的异常检测结果 | P0 |

| FR-05 | 生成可执行的节能建议 | P0 |

| FR-06 | 输出格式与系统接口契约完全匹配 | P0 |

| FR-07 | 支持全厂综合分析（多设备） | P1 |



\### 1.3 非功能需求



| 需求编号 | 需求描述 |

|---------|---------|

| NFR-01 | 工作流响应时间 ≤ 30 秒（含大模型推理） |

| NFR-02 | 支持 120 秒超时配置 |

| NFR-03 | 输出 JSON 格式符合系统契约规范 |

| NFR-04 | 代码节点具备基础容错能力 |





\## 二、 工作流整体设计



\### 2.1 设计原则



| 原则 | 说明 |

|------|------|

| \*\*数据驱动\*\* | 所有分析结论基于实际数据（statistics + data\_points），杜绝大模型凭空推测 |

| \*\*算法先行\*\* | 代码节点完成定量检测（3σ、低负载），大模型负责定性总结和建议生成 |

| \*\*契约优先\*\* | 输入输出严格遵循系统契约，确保对接零摩擦 |

| \*\*轻量高效\*\* | 代码逻辑保留核心检测能力（2 种），不引入复杂算法增加维护成本 |

\### 2.3 节点职责划分



| 节点 | 职责 | 输入 | 输出 |

|------|------|------|------|

| \*\*开始节点\*\* | 定义工作流入参 Schema | 系统 POST 的 JSON | 结构化变量供下游引用 |

| \*\*代码节点\*\* | 执行异常检测算法 | statistics, data\_points, rated\_power, device\_id, device\_name | anomalies 列表 |

| \*\*大模型节点\*\* | 生成分析摘要和节能建议 | device\_name, rated\_power, statistics, anomalies | summary + suggestions |

| \*\*结束节点\*\* | 组装完整响应并返回 | 各上游节点输出 | 符合系统契约的 JSON |

\- `data\_points` 的 Array\[Object] 需定义内部对象结构（ts, power）





\### 5.2 代码节点 — 异常检测引擎



\#### 5.2.1 节点定位



代码节点是工作流的\*\*定量分析核心\*\*，负责基于统计学方法自动检出能耗异常。它独立于大模型，确保检测结果的\*\*可复现性\*\*和\*\*确定性\*\*。



\#### 5.2.2 输入变量



| 变量名 | 类型 | 引用方式 |

|--------|------|---------|

| `device\_id` | Number | `{{start.device\_id}}` |

| `device\_name` | String | `{{start.device\_name}}` |

| `rated\_power` | Number | `{{start.rated\_power}}` |

| `statistics` | Object | `{{start.statistics}}` |

| `data\_points` | Array\[Object] | `{{start.data\_points}}` |



\#### 5.2.3 核心代码



```python

def main(statistics, data\_points, rated\_power, device\_id, device\_name):

&#x20;   """

&#x20;   能耗异常检测代码节点

&#x20;   

&#x20;   检测维度：

&#x20;     1. 3σ 统计异常检测：功率值超出均值±3倍标准差

&#x20;     2. 连续低负载检测：长时间低于额定功率×15%

&#x20;   

&#x20;   输出：

&#x20;     anomalies: Array<Object> 异常项列表

&#x20;   """

&#x20;   

&#x20;   anomalies = \[]

&#x20;   

&#x20;   # ========== 检测1：3σ 统计异常检测 ==========

&#x20;   mean = statistics.get("avg\_power", 0)

&#x20;   std = statistics.get("std\_dev", 0)

&#x20;   

&#x20;   if std > 0 and len(data\_points) > 0:

&#x20;       for point in data\_points:

&#x20;           power = point\["power"]

&#x20;           if power > mean + 3 \* std or power < mean - 3 \* std:

&#x20;               anomalies.append({

&#x20;                   "device\_id": device\_id,

&#x20;                   "device\_name": device\_name,

&#x20;                   "severity": "high",

&#x20;                   "message": f"功率异常波动：{power}kW（均值 {mean}kW，标准差 {std}kW）"

&#x20;               })

&#x20;               break  # 检出1条即退出，避免列表过长（基础功能）

&#x20;   

&#x20;   # ========== 检测2：连续低负载检测（空转识别） ==========

&#x20;   low\_threshold = rated\_power \* 0.15

&#x20;   max\_consecutive\_low = 0

&#x20;   current\_consecutive = 0

&#x20;   

&#x20;   for point in data\_points:

&#x20;       if point\["power"] < low\_threshold:

&#x20;           current\_consecutive += 1

&#x20;           if current\_consecutive > max\_consecutive\_low:

&#x20;               max\_consecutive\_low = current\_consecutive

&#x20;       else:

&#x20;           current\_consecutive = 0

&#x20;   

&#x20;   # 如果连续低负载点数超过总数据点的30%，视为空转异常

&#x20;   if len(data\_points) > 0 and max\_consecutive\_low > len(data\_points) \* 0.3:

&#x20;       anomalies.append({

&#x20;           "device\_id": device\_id,

&#x20;           "device\_name": device\_name,

&#x20;           "severity": "medium",

&#x20;           "message": f"检测到长时间低负载运行，负载率低于 {low\_threshold}kW，建议检查设备运行状态"

&#x20;       })

&#x20;   

&#x20;   return {"anomalies": anomalies}

```



\#### 5.2.4 输出变量



| 变量名 | 类型 | 说明 |

|--------|------|------|

| `anomalies` | Array\[Object] | 异常项列表，每项含 device\_id, device\_name, severity, message |



\#### 5.2.5 异常检测逻辑说明



| 检测算法 | 触发条件 | 严重等级 | 业务含义 |

|---------|---------|---------|---------|

| \*\*3σ 准则\*\* | `power > mean + 3×std` 或 `power < mean - 3×std` | `high` | 功率突发跳变或骤降，可能预示设备故障、异常停机或传感器失效 |

| \*\*连续低负载\*\* | 连续低于 `rated\_power × 15%` 的点数 > 总点数 × 30% | `medium` | 设备长时间空转或低效运行，存在节能改造空间 |





\### 5.3 大模型节点 — 报告生成引擎



\#### 5.3.1 节点定位



大模型节点负责将代码节点输出的定量检测结果转化为\*\*业务友好的自然语言报告\*\*，并生成具体的节能建议。它是工作流的\*\*定性分析核心\*\*。



\#### 5.3.2 输入变量



| 变量名 | 引用方式 |

|--------|---------|

| `device\_name` | `{{start.device\_name}}` |

| `rated\_power` | `{{start.rated\_power}}` |

| `statistics` | `{{start.statistics}}` |

| `anomalies` | `{{code.anomalies}}` |



\#### 5.3.3 Prompt 模板



```

你是一位工业能耗分析专家。请根据以下设备能耗数据生成专业的分析报告。



\## 设备信息

\- 设备名称：{{start.device\_name}}

\- 额定功率：{{start.rated\_power}} kW



\## 统计概览

\- 平均功率：{{start.statistics.avg\_power}} kW

\- 最大功率：{{start.statistics.max\_power}} kW（出现时间：{{start.statistics.max\_time}}）

\- 最小功率：{{start.statistics.min\_power}} kW（出现时间：{{start.statistics.min\_time}}）

\- 功率标准差：{{start.statistics.std\_dev}}



\## 异常检测结果

{{code.anomalies}}



\## 输出要求

请严格按以下 JSON 格式输出（不要包含任何额外文字、Markdown标记或代码块标记）：



{

&#x20; "summary": "用2-3句话总结整体能耗状况和主要问题。要求：专业、简洁、结论明确",

&#x20; "suggestions": \[

&#x20;   "建议1：具体可执行的节能措施，包含预期效果（如：建议将压力设定值从0.8MPa降至0.7MPa，预计可节能12%）",

&#x20;   "建议2",

&#x20;   "建议3"

&#x20; ]

}



注意事项：

1\. suggestions 数组包含 2-5 条建议，按推荐优先级排序

2\. 如果 anomalies 不为空，summary 必须包含异常描述

3\. 如果 anomalies 为空，summary 描述正常状态，suggestions 提供常规优化建议

4\. 建议要具体、可执行，避免空泛表述

```



\#### 5.3.4 模型配置



| 配置项 | 建议值 |

|--------|--------|

| 模型选择 | 豆包 Function Call 或 通义千问 |

| 温度 (Temperature) | 0.3（降低随机性，保证输出稳定性） |

| 输出格式 | JSON |

| Top-P | 0.8 |

| 最大输出长度 | 2048 tokens |



> 选择 \*\*温度=0.3\*\* 的原因：分析报告生成任务需要确定性输出，温度过高会导致建议内容随机变化，影响用户体验。

| `summary` | String | 分析摘要 |

\#### 5.4.1 节点定位
